§1System overview

A two-tier web application: a Vue 3 single-page application talking via HTTPS-style HTTP to a FastAPI service backed by JSON files held in memory. No database, no authentication, no persistence — operations restart from the JSON files on every server boot.

Three logical tiers; only two processes actually run (browser is the user's, server hosts API + data layer in one Python process).

What is in scope of this engagement

  • Frontend — all Vue components and composables (R1 fixes, R2 new view, D1/D2/D3)
  • Backend API — endpoints supporting the above; minor data-shape changes only
  • Test infrastructure — Playwright for E2E (R3), pytest for backend regression

What is explicitly out of scope

  • The data layer beyond JSON files — see §7 and clarifying question Q3 to procurement
  • Hosting and deployment infrastructure — supported once at handoff (W10), then operated by Meridian IT
  • Authentication, authorization, audit logging — not part of this RFP

§2Topology & processes

In development, two processes run side by side on the operator's machine. There is no separation between application server and data store.

ProcessPortStarted byStops onState
Vite dev server (Vue 3) 3000 npm run dev in client/ Ctrl+C Stateless — HMR rebuilds on file change
FastAPI / Uvicorn 8001 uv run python main.py in server/ Ctrl+C In-memory only — restart reloads JSON

Both are wrapped by scripts/start.sh for one-command boot. (Note: the script invokes python3; on Windows the binary is python — see §8 #DEBT-04.)

API documentation served live by FastAPI at http://localhost:8001/docs (Swagger UI) and http://localhost:8001/redoc.

§3Request lifecycle — a typical filtered request

The path from "operator clicks a filter" to "table re-renders" passes through five layers. Understanding it is necessary to read the R1 defects correctly: the Reports page short-circuits this path.

sequenceDiagram
    autonumber
    participant U as Operator
    participant V as Vue View
    participant F as useFilters (composable)
    participant A as api.js (Axios)
    participant S as FastAPI endpoint
    participant D as mock_data (in-memory)

    U->>V: Selects warehouse = "London"
    V->>F: writes selectedLocation.value = "London"
    Note over F: ref is module-scope —
singleton, shared across views
    F-->>V: reactive update fires
    V->>V: watch(filters) triggers reload
    V->>A: api.getInventory(getCurrentFilters())
    A->>S: GET /api/inventory?warehouse=London
    S->>S: apply_filters(items, warehouse=London, ...)
    S->>D: read inventory_items
    D-->>S: list
    S-->>A: filtered list (Pydantic models)
    A-->>V: data
    V-->>U: re-render with filtered rows
Where Reports breaks the pattern. views/Reports.vue never imports useFilters (step 2 missing), bypasses api.js (step 5 calls Axios directly), passes no query parameters (step 6 sends no filters), and has no watch (step 4 never fires). The view fetches once on mount and is permanently disconnected from the filter state. See R1 defect log #01 and #04.

§5Frontend patterns

Routing

Vue Router 4 with createWebHistory. Six routes registered in client/src/main.js:

/           → Dashboard
/inventory  → Inventory
/orders     → Orders
/demand     → Demand
/spending   → Spending
/reports    → Reports
Backlog is not routed. views/Backlog.vue exists and the backend exposes /api/backlog, but no route is registered for /backlog. The view is dead code from a routing perspective unless added to main.js. Filed as #DEBT-05 and a candidate question for Operations: was this intended to be a sub-page or surfaced from the Dashboard?

State management — composables, not Pinia/Vuex

Three composables in client/src/composables/:

  • useFilters.js — singleton: refs declared at module scope, shared across all consumers. Filter state persists across navigation by design (see §3).
  • useAuth.js — current user profile and mock tasks. Consumed by App.vue and ProfileMenu.
  • useI18n.js — translation lookup, English and Japanese locales in locales/en.js & locales/ja.js.

API access

Centralized in client/src/api.js as a single api object. All views except Reports route HTTP through this module.

Component organization

views/        page-level components, one per route
components/   shared widgets (FilterBar, modals, ProfileMenu, …)
composables/  shared reactive state & logic
locales/      i18n catalog
utils/        formatters (currency.js)

Conventions documented in client/CLAUDE.md

  • Composition API throughout — Options API discouraged
  • v-for keys must be stable IDs, never array index
  • Heavy computation in computed, not in template-called methods
  • Loading / error / data tri-state on every async view
  • Date strings parsed only after !isNaN(d.getTime())

Adherence is high across the codebase; the documented exceptions are catalogued in §8.

§6Backend patterns

Application structure

One file: server/main.py. All endpoints, Pydantic models, helper functions, and CORS middleware live in a single module. Mock data loaded from server/mock_data.py at import time.

Filter helpers

Two utilities centralize filtering logic so endpoints stay one-liners:

filter_by_month(items, month)   # supports "2025-01" and "Q1-2025"
apply_filters(items, warehouse, category, status)

The QUARTER_MAP dict at the top of main.py hardcodes 2025 quarters. New years require manual extension — flagged as #DEBT-06.

Pydantic models

One model per resource (InventoryItem, Order, DemandForecast, BacklogItem). Used as response_model on most GET routes for automatic validation and OpenAPI schema generation.

CORS

Wide-open in development (allow_origins=["*"]). Acceptable for the engagement; a hardening step before any non-local deployment.

§7Data layer

The most uncertain part of the architecture. The dev environment runs entirely on JSON files. Whether production is the same or hides a real database is the subject of clarifying question Q3 to procurement.

What is visible in the codebase

JSON fileLoaded asUsed by
server/data/inventory.jsoninventory_items/api/inventory*
server/data/orders.jsonorders/api/orders*, /api/dashboard/summary
server/data/demand_forecasts.jsondemand_forecasts/api/demand
server/data/backlog_items.jsonbacklog_items/api/backlog
server/data/spending*.jsonmultiple/api/spending/*
server/data/purchase_orders.jsonpurchase_ordersnot yet exposed

Implications

  • No persistence. Restart reloads from disk; in-flight changes during a session are lost.
  • No transactions. Concurrent writes from two clients (if it ever supported writes) would race silently.
  • No indexes. Filter helpers scan in Python; fine for the current dataset (<10K items per file), wouldn't survive 10× growth.
  • Tests written against this data layer pass against this data layer. If production has a real DB with different consistency, R3 tests need parameterization.
Action. Awaiting Q3 answer from procurement. Default assumption (per proposal/clarifying-questions.md): JSON-file layer is also production. Any divergence handled as a contract addendum.

§9Architecture decisions

Lightweight ADRs (Architecture Decision Records) for choices that will be made or are being made during this engagement. One per consequential decision; updated as the engagement proceeds.

ADR-001 — E2E test framework: Playwright

Status: Accepted Date: 2026-04-27 Phase: 1, W1

Context. R3 requires E2E coverage on critical user flows. The .mcp.json in this repo already configures a Playwright MCP server, signaling that adoption was at least considered by the previous vendor.

Decision. Adopt Playwright. Page-object pattern, deterministic assertions, CI-ready.

Consequences. Tests written against the dev environment (JSON data) — see §7 caveat. Test list (8 critical flows) defined in proposal/02-technical-approach.md §R3, refined with IT in W1.

ADR-002 — Reports remediation strategy: full rewrite, not incremental patching

Status: Proposed Date: 2026-04-27 Phase: 1, W2

Context. Reports.vue exhibits 7+ defects whose root cause is structural (Options API, no composable integration, no centralized API access). Patching them individually would leave the structural problem untouched.

Decision. Rewrite the <script> block as Composition API in a single change, integrating useFilters, useI18n, and api.js. Template adjustments (keys, computed-derived data) follow.

Consequences. Larger diff than spot fixes; mitigated by the new Playwright tests in R3 covering the Reports page (flow #4 in our test plan) which give regression confidence on the rewrite. Risk is contained — Reports is one view.

ADR-003 — Restocking recommendation logic: (s, S) policy with newsvendor calibration

Status: Proposed Date: 2026-04-27 Phase: 2, W4 — to be confirmed in Discovery

Context. R2 requires a recommendation engine. We must commit to a quantitative framework before discovery so Tanaka and her team can react to a concrete proposal, not a blank page.

Decision. Reorder point s = μ_LT + z·σ_LT from forecast and target service level; order-up-to S from forecast horizon and budget. Reference: Silver, Pyke & Thomas, Inventory and Production Management in Supply Chains, 4th ed., 2017.

Consequences. Operations must agree to the service-level parameter (typical 95–98% for industrial distribution). Operator override is a first-class feature, not an afterthought.

ADR-004 — Filter persistence across navigation: pending Operations decision

Status: Pending Date: 2026-04-27 Phase: 1, W1 — Tanaka working session

Context. useFilters uses a module-scope singleton. Filter state persists across navigation. The previous vendor's intent is unrecoverable.

Decision. Defer to R. Tanaka in the W1 working session. Three options on the table: confirm as-is, reset on navigation, or surface active filters in the header with a one-click reset.

Consequences. The chosen path is small in implementation — at most a refactor of useFilters.js plus a status indicator. Clarification matters more than the code.

ADR-005 — Production data layer: assumed JSON until refuted

Status: Provisional Date: 2026-04-27 Phase: 1, pending Q3 from procurement

Context. The codebase has no database. Production may or may not have one — Q3 in proposal/clarifying-questions.md.

Decision. Until told otherwise, assume the JSON-file layer is also production. R3 tests, R4 documentation, and pricing reflect this. Any divergence triggers a contract addendum, not silent absorption.

Consequences. If a database is revealed late, schedule extends 1–2 weeks (per 03-timeline.md) and pricing is re-scoped.
